# Review Skill — 評測範例

## 好的輸出範例

> 題目：審查一份「越南 AI 工具調查報告」

```markdown
# 審查報告：vietnam-ai-tools-research.md

## 總體評估
有條件通過 — 結構完整、結論有依據，但有 2 處事實待確認。

## 通過項
- 四區塊分類清楚（已知事實 / 合理推論 / 待驗證 / 高風險假設）
- 結論先行，第一段即摘要
- 來源標示完整（5 個來源，含日期）

## 必須修改
- 【已知事實第 3 點】「越南 AI 市場規模達 US$2B」— 來源為 2024 年報告，
  2026 年數據可能已變動。建議：標記為 [待驗證] 或補上 2026 年來源。
- 【合理推論第 1 點】「本地工具將在 2 年內取代國際工具」— 推論跳躍過大，
  缺乏中間論證。建議：補上支持此推論的具體趨勢數據，或降級為高風險假設。

## 建議修改
- 【結論段】可加入一句話說明調查的限制（如：僅涵蓋胡志明市企業）
- 【來源區】建議統一格式為「作者, 標題, 日期, URL」

## Definition of Done 逐條確認
- [x] 涵蓋至少 5 個工具的市場數據 — 通過（列出 6 個）
- [x] 包含本地 vs 國際工具比較 — 通過（第三節有比較表）
- [x] 所有事實標明來源 — 通過
- [ ] 推論與假設明確標記 — 部分通過（第 1 點推論過強，見必須修改）
- [x] Markdown 格式 — 通過
```

**為什麼好**：問題具體到段落位置；區分必須修改 vs 建議修改；逐條比對 definition_of_done；有指出做得好的地方。

## 壞的輸出範例

> 同一題目的錯誤輸出

```markdown
# 審查報告

## 評估
整體寫得不錯，內容豐富，結構清晰。建議可以再加強一些細節。

## 問題
- 有些地方可以更具體
- 來源可以更多
- 格式可以再調整

## 結論
通過。
```

**哪裡錯**：
- 問題不具體（「有些地方」是哪裡？）
- 沒有區分必須修改 vs 建議修改
- 沒有逐條比對 definition_of_done
- 審查太寬鬆，直接說「通過」但沒有驗證依據
- 缺少通過項（只挑毛病，沒指出做得好的地方）

---

## 判斷標準

| 項目 | 通過 | 不通過 |
|------|------|--------|
| 具體性 | 問題定位到段落/句子 | 「有些地方」「部分內容」等模糊描述 |
| 分級 | 必須修改 vs 建議修改明確區分 | 所有問題混在一起 |
| DoD 比對 | 逐條列出 + 通過/未通過 | 沒有比對或只說「大致符合」 |
| 平衡 | 有通過項 + 問題項 | 只挑毛病或全部說好 |
| 嚴重度 | 嚴重問題放在必須修改 | 嚴重問題被降級為建議 |
